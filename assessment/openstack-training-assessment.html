Repo Assessment

OpenStack Training Repo Health Check

This assessment covers the executable lesson assets across the training repository, with special focus on the Ubuntu-compatible OpenStack path in day4, the role-driven material in day5, and the advanced playbook lessons in Day6.

The repo is now in a materially stronger state: parser failures were removed, role resolution was normalized, the OpenStack installer now uses a project-local virtual environment, and the DevStack cluster scripts were hardened for Ubuntu/LXD usage.

Before

Initial Failures

  • scenario0_prerequisites.yml failed syntax checks because a debug message block was malformed.
  • day5/ansible-tower-awx/playbooks/10-deploy-everything-single-instance.yml and 02-install-awx-openstack.yml used unescaped Docker Go templates that Ansible parsed as broken YAML/Jinja.
  • day5/ansible-roles-galaxy/roles/beginner/database/vars/RedHat.yml started with invalid text and broke YAML parsing.
  • scenario9_roles_multivm.yml referenced a role that did not exist.
  • Several teardown playbooks called openstack.cloud.router_interface, which is not available in the installed openstack.cloud collection.
  • The OpenStack install scripts relied on direct pip install --user, which is brittle on Ubuntu 24.04 due to system-managed Python packaging.
  • deploy_three_devstacks.sh built remote shell commands with literal \n sequences, creating runtime breakage inside containers.
  • deploy_three_devstack_instances.sh contained a stray EOF path and was missing important nested-container runtime settings for DevStack.
After

What Was Fixed

  • The broken YAML and AWX playbook parser issues were corrected, and all checked executable playbooks now pass syntax validation.
  • A shared root ansible.cfg now defines roles_path, which makes the in-repo lesson roles discoverable when the repo is run from its root.
  • A real roles/vm_deployment role was added so Scenario 9 is now a functioning role-based lesson instead of a missing stub.
  • Router teardown steps were updated to use the OpenStack CLI, matching the installed tooling and removing unsupported module usage.
  • The day4 OpenStack installers now use a local .venv, work from any directory, and were validated by actually creating the environment and verifying the SDK plus collection installation.
  • The DevStack cluster scripts were hardened for Ubuntu: container nesting/privileged settings were added, kvm passthrough is attempted when available, and a generated clouds file is produced for the triple-stack workflow.
  • Fact-independent timestamps replaced ansible_date_time in places where the lessons were running with gather_facts: false.

Environment Notes

The code was assessed from this workspace with Ansible and Python available locally, but without a live Ubuntu LXD host that could actually launch and finish a triple DevStack/OpenStack deployment. The cluster automation scripts were shell-validated and hardened specifically for that Ubuntu path, but the full control-plane bring-up still needs to be executed on a real Ubuntu host with LXD, nested virtualization support, and sufficient RAM/disk.

That means the repo is now in a much better state to perform that deployment, but the final proof of a working full OpenStack cluster still depends on running the cluster scripts in the correct target environment.
